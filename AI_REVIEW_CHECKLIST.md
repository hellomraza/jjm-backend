## 1. General Checks

Before committing code, verify:

- [ ] Only **relevant files** were modified — no unrelated changes
- [ ] File change count is within **allowed limit**
- [ ] Commit message follows **standard format** (e.g., `feat:`, `fix:`, `refactor:`)
- [ ] No **debugging code** remains (`debugger`, test logs, etc.)
- [ ] No **`console.log`** statements exist — use `Logger` instead
- [ ] No **commented-out code** left behind
- [ ] No **TODO/FIXME** comments without corresponding issue tracking

### Commit Message Format:

```
<type>(<scope>): <short description>

Types: feat | fix | refactor | docs | test | chore
Scope: module name (e.g., auth, projects, uploads)

Examples:
  feat(auth): add JWT refresh token endpoint
  fix(uploads): validate file size before S3 upload
  refactor(projects): extract approval logic to service
```

## 2. Architecture Checks

Verify NestJS module structure compliance:

- [ ] Code follows **modular architecture** (`module → controller → service → entity`)
- [ ] **Controllers** contain only routing logic — no business logic
- [ ] **Services** contain all business logic — no request/response objects
- [ ] **DTO validation** exists for every request body
- [ ] **Entities** match the database schema accurately
- [ ] File is placed in the **correct module directory**
- [ ] Module **imports and providers** are registered in `*.module.ts`

### Expected Structure:

```
src/
  module-name/
    module-name.module.ts        ← registers providers, imports
    module-name.controller.ts    ← routing only
    module-name.service.ts       ← business logic
    dto/
      create-module-name.dto.ts  ← validated with class-validator
      update-module-name.dto.ts  ← extends PartialType(CreateDto)
    entities/
      module-name.entity.ts      ← TypeORM entity
```

## 3. Security Checks

Verify security compliance:

### Authentication:

- [ ] Passwords are hashed with **bcrypt** (min 10 salt rounds)
- [ ] Password field is **never returned** in any API response
- [ ] **JWT authentication** is applied via `JwtAuthGuard`
- [ ] JWT secret is stored in **environment variables** (not hardcoded)
- [ ] Token has proper **expiration** configured

### Endpoint Protection:

- [ ] **Role guard** (`RolesGuard`) protects sensitive endpoints
- [ ] All sensitive endpoints **require authentication**
- [ ] No endpoint exposes **internal system data** (stack traces, DB errors)

### Data Protection:

- [ ] Sensitive fields are **excluded from responses** (`password`, `token`, etc.)
- [ ] User input is **sanitized** before processing
- [ ] No **SQL injection** vectors exist (using TypeORM repositories, not raw SQL)

## 4. Validation Checks

Verify input validation:

- [ ] **DTO validation** exists for every `POST` / `PATCH` endpoint
- [ ] **`class-validator`** decorators are used (`@IsString`, `@IsEmail`, `@IsNotEmpty`, etc.)
- [ ] Invalid requests are **rejected** with `BadRequestException`
- [ ] Missing required fields **throw meaningful errors**
- [ ] `ValidationPipe` is applied **globally or per-route**
- [ ] `Update*Dto` extends `PartialType(Create*Dto)`

### Example Validation:

```typescript
// create-project.dto.ts
import { IsString, IsNotEmpty, IsNumber } from 'class-validator';

export class CreateProjectDto {
  @IsString()
  @IsNotEmpty()
  name: string;

  @IsNumber()
  district_id: number;

  @IsNumber()
  contractor_id: number;
}
```

## 5. Database Checks

Verify database usage:

- [ ] Entities use correct **TypeORM decorators** (`@Entity`, `@Column`, `@PrimaryGeneratedColumn`)
- [ ] **Repository pattern** is used for all data access
- [ ] **No raw SQL** unless absolutely necessary (and documented with reason)
- [ ] **Transactions** are used for critical multi-step operations
- [ ] **Relations** are properly defined (`@ManyToOne`, `@OneToMany`, `@JoinColumn`)
- [ ] `created_at` and `updated_at` columns exist on all entities
- [ ] **Migrations** are used for schema changes (not `synchronize: true` in production)

### Transaction Example:

```typescript
async approveComponent(id: number): Promise<void> {
```

## 6. Photo Upload Checks

Verify file upload logic:

### Upload Pipeline:

- [ ] **Multer** is used via `FileInterceptor` / `FilesInterceptor`
- [ ] File **type is validated** (allowed: `jpg`, `png`, `jpeg`, `pdf`)
- [ ] File **size is validated** (max: **5 MB**)
- [ ] File is uploaded to **AWS S3** (not stored locally)
- [ ] **Metadata is saved** to the database after successful upload

### Required Metadata Fields:

| Field          | Type       | Required | Description                      |
|----------------|------------|----------|----------------------------------|
| `latitude`     | `decimal`  | Yes      | GPS latitude of photo location   |
| `longitude`    | `decimal`  | Yes      | GPS longitude of photo location  |
| `timestamp`    | `datetime` | Yes      | Time the photo was captured      |
| `employeeId`   | `number`   | Yes      | ID of the uploading employee     |
| `componentId`  | `number`   | Yes      | ID of the related component      |
| `workOrderId`  | `number`   | Yes      | ID of the associated work order  |

> **If any required metadata field is missing, reject the upload** with `BadRequestException`.

### Upload Validation Example:

```typescript
@Post('upload')
@UseGuards(JwtAuthGuard, RolesGuard)
@Roles('EM')
@UseInterceptors(FileInterceptor('file', {
  limits: { fileSize: 5 * 1024 * 1024 },  // 5 MB
  fileFilter: (req, file, cb) => {
    if (!file.mimetype.match(/\/(jpg|jpeg|png|pdf)$/)) {
      cb(new BadRequestException('Unsupported file type'), false);
    }
    cb(null, true);
  },
}))
async uploadPhoto(
  @UploadedFile() file: Express.Multer.File,
  @Body() dto: UploadPhotoDto,
) {
  return this.uploadService.upload(file, dto);
}
```

## 8. Component Approval Rule

### Sequential Approval Enforcement:

> **Component N+1 CANNOT start before Component N is approved.**

- [ ] Service logic **enforces sequential ordering** of component approvals
- [ ] Attempting to start/approve a later component **throws an error** if the previous one is not approved
- [ ] The check is performed in the **service layer** (not just frontend validation)
- [ ] Edge cases are handled (first component, already approved, etc.)

### Implementation Example:

```typescript
async startComponent(workOrderId: number, componentNumber: number): Promise<void> {
  if (componentNumber > 1) {
    const previousComponent = await this.componentRepo.findOne({
      where: { workOrderId, componentNumber: componentNumber - 1 },
    });

    if (!previousComponent || previousComponent.status !== 'APPROVED') {
      throw new BadRequestException(
        `Component ${componentNumber - 1} must be approved before starting Component ${componentNumber}`,
      );
    }
  }

  // proceed with starting the component
}
```

## 9. Race Condition Checks

Verify concurrency safety for critical operations:

- [ ] **Database transactions** are used for approval/status-change operations
- [ ] **Row-level locking** is used to prevent concurrent modifications (`FOR UPDATE`)
- [ ] **Duplicate approvals** are prevented (check current status before updating)
- [ ] **Optimistic or pessimistic locking** is applied where needed

### Race-Safe Approval Example:

```typescript
async approveComponent(componentId: number): Promise<void> {
  const queryRunner = this.dataSource.createQueryRunner();
  await queryRunner.connect();
  await queryRunner.startTransaction();

  try {
    const component = await queryRunner.manager.findOne(Component, {
      where: { id: componentId },
      lock: { mode: 'pessimistic_write' },  // row lock
    });

    if (!component) {
      throw new NotFoundException('Component not found');
    }

    if (component.status === 'APPROVED') {
      throw new ConflictException('Component already approved');
    }

    component.status = 'APPROVED';
    component.approvedAt = new Date();
    await queryRunner.manager.save(component);
    await queryRunner.commitTransaction();
  } catch (err) {
    await queryRunner.rollbackTransaction();
    throw err;
  } finally {
    await queryRunner.release();
  }
}
```

## 11. Response Structure Checks

All API responses **must** follow a consistent structure.

### Standard Success Response:

```json
{
  "success": true,
  "message": "Operation completed successfully",
  "data": { }
}
```

### Standard Error Response:

```json
{
  "success": false,
  "message": "Descriptive error message",
  "statusCode": 400
}
```

### Response MUST NOT contain:

| Forbidden Field  | Reason                              |
|------------------|-------------------------------------|
| `password`       | Security — never expose credentials |
| `internal IDs`   | Security — prevent enumeration      |
| `stack traces`   | Security — hide implementation details |
| `raw DB errors`  | Security — prevent information leaks |
| `JWT secrets`    | Security — never expose tokens/keys |

- [ ] Response follows the **standard structure** above
- [ ] Sensitive fields are **excluded**
- [ ] Error responses use **NestJS HTTP exceptions** with meaningful messages
- [ ] List endpoints return **paginated data** with `total`, `page`, `limit`
